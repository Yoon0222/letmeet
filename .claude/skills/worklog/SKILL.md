# 워크로그 (피클 작업 일지)

`docs/WORKLOG.md` 하나에 **핵심만** 계속 쌓는다. 장황한 대화 전문이 아니라, 나중에 봐도 맥락이 복원되는 **결정 기록(decision log)** 이다.

## 언제 갱신하나
- 기능/화면/DB 변경을 **완료**했을 때
- 방향을 바꾸는 **결정**을 내렸을 때 (예: 탭 의미 변경, 스택 선택)
- 사용자가 "기록/로그 정리해줘" 라고 할 때
- 세션을 마무리할 때

## 절차
1. `docs/WORKLOG.md` 를 읽는다. (없으면 아래 구조로 생성)
2. **직전 엔트리 이후** 일어난 일만 요약해 새 하위 엔트리(`### 제목`)를 만든다.
   - **오늘 날짜 섹션(`## YYYY-MM-DD`)이 이미 있으면** 그 섹션 안 최상단에 하위 엔트리를 추가한다. (날짜 헤더를 새로 찍지 말 것 — 같은 날짜 중복 금지)
   - **없으면** 엔트리 영역 최상단에 `## 오늘날짜` 섹션을 새로 만들고 그 아래 첫 엔트리를 넣는다.
3. 상단 "## 열린 항목" 체크리스트를 **최신화**한다 (끝난 건 [x], 새로 생긴 건 추가).
4. 날짜는 상대표현 금지 — 절대날짜(YYYY-MM-DD)로. (오늘 날짜는 세션 컨텍스트의 currentDate 사용)

## 파일 구조
```markdown
# 피클 — 작업 일지 (WORKLOG)

## 열린 항목
- [ ] 아직 안 끝난 할 일 (누가/무엇)
- [x] 끝난 건 체크

---

## YYYY-MM-DD        ← 하루당 헤더 하나. 최신 날짜가 위로.

### <한 줄 제목>      ← 그날의 작업 하나. 같은 날 새 작업은 이 섹션 안 위쪽에 추가.
- **결정**: 무엇을 왜 그렇게 정했는지 (1~3줄)
- **만든 것**: 파일/기능 (핵심만, 경로 포함)
- **메모/주의**: 함정, 후속 필요 (선택)

### <또 다른 제목>
- ...
```

## 작성 규칙
- 엔트리는 **6줄 이내**를 목표로. 코드 diff 를 붙이지 말고 "무엇을/왜"만.
- 결정의 **이유**를 꼭 남긴다 (나중에 "왜 이렇게 했지?" 를 없앤다).
- 파일 언급은 경로로 (`src/app/(tabs)/clubs.tsx`).
- 완료된 열린 항목은 지우지 말고 [x] 로 남겨 히스토리를 유지.
- 노션 연동이 켜지면 이 파일을 노션 페이지에도 미러링할 수 있다(마크다운 그대로 붙여넣기 가능).
